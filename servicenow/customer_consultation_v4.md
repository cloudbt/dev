# SG-SCCM Data Source 修正に関するご相談

**2026年4月21日 / SGCチーム**

---

## スライド1：背景と対応方針

### 発生事象
SG-SCCM マルチインスタンス構成（一般系 `it_prod` / 放送系 `ot_prodE`）において、**一般系の Import 実行時に、放送系の CI レコードが誤って更新される事象**を確認しました。

具体的には、放送系端末の Name や Serial number が一般系端末の値で上書きされ、**同一の Name / Serial を持つ CI が2件存在する状態**が発生しています。

> **発生例**
> - 放送系CI（端末A）：Name=`wrmas-f-ost03` / Serial=`8Z018G4`
> - 一般系CI（端末B）：Name=`01620-21-p07` / Serial=`1117Q115H`
>
> ① 一般系 Import 実行 → 端末Aの値が端末Bの値で上書きされる
> ② 放送系 Import 実行 → 端末Aが元の値に復元
> ③ 以降 ①② が繰り返し発生し、CI 属性の履歴が汚染される

### ServiceNow サポート回答
- **原因**：両 Data Source の SQL で `u_connectionid` が同一値となっており、`sys_object_source` ID が重複 → IRE が誤マッチ
- **対応**：両 Data Source の `u_connectionid` を異なる値に修正
- **注意事項**：修正後は全 CI が新規 INSERT として処理される可能性あり。**本番適用前に検証環境での動作確認が必須**

### 課題
当方検証環境は **放送系のみ接続** のため、両系統のデータが揃わず本事象の再現・検証ができない。

## スライド2：2つの選択肢

| 観点 | 案A：検証環境から本番MCMへ一時接続<br>（テスト後データ削除） | 案B：検証をスキップし本番直接対応 |
|---|---|---|
| 事象再現・事前検証 | ◎ 可能 | × 不可 |
| 本番への影響リスク | 低 | **高（ぶっつけ対応）** |
| ServiceNow 推奨事項の遵守 | ◎ | × |
| 想定外事象発生時の復旧性 | 検証で手順確立済 | 本番で即興対応 |
| お客様側ご調整 | 本番MCMへの一時接続許可 | 本番作業立会い等 |

※ 案A参考：Tanium Discovery 連携では既に検証環境から本番接続を許可いただいている前例あり

---

## スライド3：推奨案とご判断事項

### 推奨：**案A（検証環境から本番 MCM への一時接続）**

**理由**
1. ServiceNow サポートが検証環境での事前確認を明確に推奨
2. CI 誤更新事象は構成管理情報の信頼性に直結するため、修正時の追加トラブルは回避したい
3. 案B は本番作業中の想定外事象に対し、影響範囲の予測・即時復旧判断が困難

### ご判断いただきたい事項
- 案A / 案B のいずれで進めるか
- 案A の場合：検証環境から本番 MCM への一時接続をご許可いただけるか
  （接続期間・データ削除条件等はご要望に応じて調整）
