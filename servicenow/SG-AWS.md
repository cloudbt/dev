## ■ 背景・目的

仮想マシン等のクラウドリソースについて、自動収集以外にも手動登録が可能であることを前提とし、ポータル画面でCIの登録・更新申請を実施する際に、運用面で考えられる懸念点を整理し、iNOC担当者へ引き継ぐことを目的とする。

## ■ 前提条件

  * 対象は AWS / Azure 上の仮想マシン等のクラウドリソースとする。
  * Service Graph Connector for Azure/AWS による自動収集は利用せず、CIの登録・更新はポータル画面からの申請を起点に手動で実施する前提とする。

## ■ 手動登録運用における懸念点

| No | 懸念事項 | 懸念内容詳細 | 備考 |
| :--- | :--- | :--- | :--- |
| 1 | 誤登録時のメンテナンス負荷 | 誤ってクラウドリソースを登録した際は、iNOC担当者を通じたメンテナンスが必要となる可能性がある。<br>修正依頼フロー、確認観点、責任分界を事前に整理しておく必要がある。 | |
| 2 | CI Relationships が収集されない | 依存関係ビューは作成されないため、図で俯瞰してCI関係を見るということができなくなる。<br>→CI Relationshipsが収集されないため、各CIの関係性がわからないため、障害時の影響範囲確認や構成把握が難しくなる懸念がある。 | |
| 3 | リージョン関係を維持した登録が難しい | クラウドアカウント内のリソースについてリージョン関係を維持して登録ができない。<br>→自動収集の場合はアカウントに紐づく形でリージョン別にリソースが登録されるが、手動登録の場合アカウントに紐づける形でリージョン別にリソースを登録することが難しい。 | |
| 4 | アプリケーションサービスとの紐づけ基準整理が必要 | 手動登録CIをアプリケーションサービスにどの単位で紐づけるかの方針決定が必要。<br>パターン①：アカウント単位（リソース確認容易、関係性不明）<br>パターン②：EC2単位（関係性明確、申請量膨大）<br>パターン③：アプリケーションサービスに親子関係を作成し、子アプリケーションサービスにEC2単位で紐づける（自動収集に近い管理、構造複雑） | 案件ごとに登録方針がぶれないよう、関係者間での合意が必要である。 |
| 5 | VR/SIR・設備種別・管理グループへの影響 | クラウドサービスアカウントの設備種別・管理グループ情報はビジネスルール（BR）により配下リソースに自動反映されるが、手動登録のクラウドリソースがクラウドサービスアカウントと正しく紐づかない場合、VRやSIRのモジュールでの脆弱性管理やセキュリティインシデント対応にCIが正しく関連付けられず、対応漏れ・対応遅延のリスクがある。 | 手動登録CIとクラウドサービスアカウントの紐づけ方法を確立する。<br>例：「親クラウドサービスアカウント」の指定を必須項目として必要。 |
| 7 | 登録対象範囲のばらつき | 仮想マシンのみを登録対象とするのか、ネットワーク・ストレージ・セキュリティ関連CIまで含めるのかが明確でないと、案件ごとに登録粒度が変わってしまう。登録対象の範囲と標準粒度を事前に定義しておく必要がある。 | |
| 8 | 自動収集CIと手動登録CIの混在管理 | 一部アカウントが自動収集、一部が手動登録という混在状態では、同一CMDB内で管理方針が二重化する。CI検索・レポーティング・変更管理で収集元による挙動差異を意識した運用が必要。Discovery Source属性やタグでの識別ルールを定めるべき。 | 手動登録CIをDiscovery Source属性またはタグで識別可能にする。 |

※No.1～4は2024/11/6(水)構成管理分科会でNHK様に共有済みの懸念点です。

## ■ まとめ

仮想マシン等のクラウドリソースについて、手動登録そのものは可能と考えられるが、自動収集を利用する場合と比べると、CI関係性の不足、リージョンや階層構造の維持困難、紐づけ基準の整理不足、属性反映やVR/SIRへの影響、更新追随漏れ、登録対象範囲の不統一、自動収集CIと手動登録CIの混在管理といった懸念がある。

そのため、ポータル画面からCI登録・更新申請を前提に運用する場合は、単に登録できるかどうかではなく、『どの粒度で、どの基準で、誰が、どのタイミングで維持管理するか』を明確にした上で運用設計する必要がある。
