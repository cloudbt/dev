## ■ 背景・目的

仮想マシン等のクラウドリソースについて、自動収集以外にも手動登録が可能であることを前提とし、ポータル画面でCIの登録・更新申請を実施する際に、運用面で考えられる懸念点を整理し、iNOC担当者へ引き継ぐことを目的とする。

## ■ 前提条件

  * 対象は AWS / Azure 上の仮想マシン等のクラウドリソースとする。
  * Service Graph Connector for Azure/AWS による自動収集は利用せず、CIの登録・更新はポータル画面からの申請を起点に手動で実施する前提とする。

## ■ 手動登録運用における懸念点

## ■ まとめ

仮想マシン等のクラウドリソースについて、手動登録そのものは可能と考えられるが、自動収集を利用する場合と比べると、CI関係性の不足、リージョンや階層構造の維持困難、紐づけ基準の整理不足、属性反映やVR/SIRへの影響、更新追随漏れ、登録対象範囲の不統一、自動収集CIと手動登録CIの混在管理といった懸念がある。

そのため、ポータル画面からCI登録・更新申請を前提に運用する場合は、単に登録できるかどうかではなく、『どの粒度で、どの基準で、誰が、どのタイミングで維持管理するか』を明確にした上で運用設計する必要がある。

# クラウドリソースを手動で登録する際の注意事項

## ■ 背景・目的

NHK様環境では、AWS / Azure 上のクラウドリソース（仮想マシン等）の構成情報は、Service Graph Connector for AWS（SG-AWS）/ Azure による自動収集を標準運用としている。

ただし、以下のようなケースでは自動収集の対象外となり、ポータル画面からの手動登録が必要となる場合がある。

- SG-AWS / SG-Azure の収集対象外となるリソース種別
- SG-AWS / SG-Azure が未導入のアカウントに属するリソース
- テスト環境や一時的に作成されたリソース

本セクションでは、手動登録を行う際にiNOC担当者が注意すべきAWS / Azure固有の考慮事項と、その対処方法を整理する。

> **手順の所在：** 手動登録の具体的な操作手順は利用者向け手順書「〇〇」を参照のこと。本セクションは、手動登録時にAWS / Azure固有で追加考慮が必要な事項に限定して記載する。

## ■ 前提条件

1. SG-AWS / SG-Azure による自動収集が標準運用であり、手動登録は自動収集でカバーできない場合の補完手段として位置づける。
2. 手動登録の対象リソース種別は以下とする。
   - 【要確認】対象機種（Windows VM、Linux VM、NW機器等）を明記する ※Leader確認事項①
3. 通常のCI登録手順（ポータル画面からの変更申請）で対応可能な範囲はそのまま適用し、本セクションではAWS / Azure固有で追加考慮が必要な事項のみ記載する。
4. 自動収集CIと手動登録CIはCMDB内で混在する。混在環境での識別方法は後述の「混在管理時の識別・フィルタ方法」を参照のこと。

## ■ AWS / Azure 固有の注意事項と対処方法

### 1. 誤登録時の是正方法

| 項目 | 内容 |
| :--- | :--- |
| **事象** | 手動登録時に誤った情報でCIを登録してしまった場合、是正が必要となる。 |
| **通常CIとの違い** | 通常のCI（オンプレミス機器等）は変更申請により是正可能である。AWS / Azureのクラウドリソースについても、変更申請で是正可能な項目と、iNOC担当者による直接修正が必要な項目がある。【要確認】変更申請で是正できない項目があるか ※Leader確認事項② |
| **対処方法** | 変更申請で是正可能な場合：通常の変更申請フローに従い是正する。<br>変更申請で是正できない項目がある場合：iNOC担当者にて直接修正を実施する。修正対象項目と手順は「〇〇」を参照。 |

### 2. CI Relationships（依存関係）が自動作成されない

| 項目 | 内容 |
| :--- | :--- |
| **事象** | 手動登録したクラウドリソースでは、依存関係ビュー（CI Relationships）が自動作成されない。 |
| **理由** | SG-AWS / SG-Azure による自動収集では、APIを通じてリソース間の依存関係情報を取得し、CI Relationshipsとして自動登録する。手動登録ではこのAPI取得プロセスが介在しないため、Relationshipsは作成されない。 |
| **影響** | 障害時の影響範囲確認や構成全体の俯瞰が、依存関係ビューからは行えない。 |
| **対処方法** | 【要確認】手動でCI Relationshipsを作成する手順が存在するか確認の上、手順を記載する。手順がない場合は、本制約を受容した上で、影響範囲確認は別途管理台帳等で補完する運用とする。 ※Leader確認事項③ |

### 3. リージョン・アカウント階層構造の登録

| 項目 | 内容 |
| :--- | :--- |
| **事象** | 自動収集では、クラウドサービスアカウント配下にリージョン別でリソースが階層的に登録されるが、手動登録ではこの階層構造が自動的には反映されない。 |
| **理由** | SG-AWS / SG-Azure はAPI応答に含まれるアカウントID・リージョン情報をもとに階層構造を構築する。手動登録ではこれらの情報を利用者が明示的に入力する必要がある。 |
| **対処方法** | 手動登録時に以下の項目を必須入力とすることで、階層構造を維持する。<br>- **親クラウドサービスアカウント**（Hosted on 関係の指定）<br>- **リージョン情報**（該当する属性フィールドへの入力）<br>具体的な入力手順は利用者向け手順書に追記する。【要確認】ポータル申請フォームでこれらの項目が入力可能か検証が必要 ※Leader確認事項④ |

### 4. アプリケーションサービスとの紐づけ

| 項目 | 内容 |
| :--- | :--- |
| **事象** | 手動登録CIをアプリケーションサービスにどの単位で紐づけるかの方針が必要。 |
| **選択肢と条件分岐** | 以下の条件に基づき紐づけパターンを選択する。<br><br>**条件A：アカウント内のリソース数が少なく、業務影響管理の粒度が粗くてよい場合**<br>→ **パターン①：アカウント単位で紐づけ**<br>　メリット：管理が簡易、申請量が少ない<br>　制約：個別リソースの業務影響が不明確<br><br>**条件B：個別リソースごとの業務影響管理が必要な場合**<br>→ **パターン②：EC2等リソース単位で紐づけ**<br>　メリット：関係性が明確、障害時の影響特定が容易<br>　制約：申請量が多くなる<br><br>**条件C：自動収集に近い管理粒度が求められる場合**<br>→ **パターン③：親子アプリケーションサービスを作成し、子にリソース単位で紐づけ**<br>　メリット：自動収集と同等の管理が可能<br>　制約：構造が複雑、設計・維持コストが高い |
| **ISM側推奨** | 【要確認】ISM側としての推奨パターンまたはデフォルト方針を決定する必要あり ※Leader確認事項⑤ |
| **手順** | 決定した方針に基づく紐づけ手順を利用者向け手順書に追記する。 |

### 5. クラウドサービスアカウントとの紐づけ（VR/SIR影響）

| 項目 | 内容 |
| :--- | :--- |
| **事象** | 手動登録のクラウドリソースがクラウドサービスアカウントと正しく紐づかない場合、設備種別・管理グループ情報がビジネスルール（BR）により自動反映されず、VR（脆弱性管理）やSIR（セキュリティインシデント対応）のモジュールでCIが正しく関連付けられないリスクがある。 |
| **対処方法** | 手動登録時に**「親クラウドサービスアカウント」を必須入力項目**とする。<br>これにより、BRによる設備種別・管理グループの自動反映が機能し、VR/SIRでの正しい関連付けが担保される。<br><br>**具体的な手順：**<br>1. 登録対象リソースが属するクラウドサービスアカウントのCI名を確認する<br>2. ポータル申請フォームの「親クラウドサービスアカウント」フィールドに上記CI名を指定する<br>3. 登録後、設備種別・管理グループが正しく反映されていることを確認する<br><br>【要確認】上記手順でBRが正しく動作するか、実機検証が必要 ※Leader確認事項⑥ |

### 6. 登録対象範囲の定義

| 項目 | 内容 |
| :--- | :--- |
| **事象** | 手動登録の対象をどのリソース種別まで含めるかが不明確だと、案件ごとに登録粒度がばらつく。 |
| **ISM側の定義** | 手動登録の標準対象範囲を以下のとおり定義する。【要確認】以下の定義で妥当か ※Leader確認事項⑦<br><br>- **登録対象：** 仮想マシン（EC2 / Azure VM）<br>- **登録対象外（自動収集でカバー）：** VPC、サブネット、セキュリティグループ、ストレージ（S3 / Blob）等のネットワーク・ストレージ・セキュリティ関連リソース<br>- **判断が必要な場合：** 上記に該当しないリソース種別については、iNOC担当者からISMチームへ確認の上、登録要否を判断する。 |

## ■ 混在管理時の識別・フィルタ方法

自動収集CIと手動登録CIがCMDB内で混在することは前提条件となる。以下の方法で識別・フィルタを行う。

| 操作 | 方法 |
| :--- | :--- |
| **識別方法** | Discovery Source属性により識別する。<br>- 自動収集CI：Discovery Source = "ServiceGraph" 等（SG-AWS/Azureが自動設定）<br>- 手動登録CI：Discovery Source = "Manual" または空欄（手動登録時に "Manual" を設定するルールとする） |
| **CI検索時のフィルタ** | CI一覧画面にて、Discovery Sourceカラムでフィルタを適用する。<br>例：手動登録CIのみ表示 → Discovery Source = "Manual" で絞り込み |
| **レポート作成時** | レポート条件にDiscovery Sourceを含め、自動収集CI/手動登録CIを分けて集計可能とする。 |

## ■ まとめ

AWS / Azureのクラウドリソースの手動登録について、ISM側で以下の対処を行っている。

**ISM側で技術的にカバーする範囲：**
- 手動登録時の必須入力項目の定義（親クラウドサービスアカウント、リージョン情報）
- クラウドサービスアカウントとの紐づけ手順（VR/SIR正常動作の担保）
- 登録対象リソース種別の標準範囲の定義
- 自動収集CI/手動登録CIの識別ルール（Discovery Source属性）とフィルタ手順
- 紐づけパターンの条件分岐と各パターンの手順（利用者向け手順書に追記）

**技術的にカバーできない範囲（各構築案件に依存）：**
- CI Relationships（依存関係）は手動登録では自動作成されないため、障害時の影響範囲確認は別途管理台帳等で補完が必要となる。
- アプリケーションサービスとの紐づけパターンの最終選定は、案件の業務要件に依存するため、案件ごとに判断が必要。ISM側では条件分岐の基準と各パターンの手順を提供する。
- 案件固有の管理要件（特殊なタグ付け、独自の分類等）については、汎用項目（Attributes等）を活用して案件ごとに対応いただきたい。

# クラウドリソースを手動で登録する際の注意事項

## ■ 背景・目的

NHK様環境では、AWS / Azure 上のクラウドリソース（仮想マシン等）の構成情報は、Service Graph Connector for AWS（SG-AWS）/ Azure による自動収集を標準運用としている。

ただし、以下のようなケースでは自動収集の対象外となり、ポータル画面からの手動登録が必要となる場合がある。

- SG-AWS / SG-Azure の収集対象外となるリソース種別
- SG-AWS / SG-Azure が未導入のアカウントに属するリソース
- テスト環境や一時的に作成されたリソース

本セクションでは、手動登録を行う際にiNOC担当者が注意すべきAWS / Azure固有の考慮事項と、その対処方法を整理する。

> **手順の所在：** 手動登録の具体的な操作手順は利用者向け手順書「〇〇」を参照のこと。本セクションは、手動登録時にAWS / Azure固有で追加考慮が必要な事項に限定して記載する。

## ■ 前提条件

1. SG-AWS / SG-Azure による自動収集が標準運用であり、手動登録は自動収集でカバーできない場合の補完手段として位置づける。
2. 手動登録の対象リソース種別は以下とする。
   - 【要確認】対象機種（Windows VM、Linux VM、NW機器等）を明記する ※Leader確認事項①
3. 通常のCI登録手順（ポータル画面からの変更申請）で対応可能な範囲はそのまま適用し、本セクションではAWS / Azure固有で追加考慮が必要な事項のみ記載する。
4. 自動収集CIと手動登録CIはCMDB内で混在する。混在環境での識別方法は後述の「混在管理時の識別・フィルタ方法」を参照のこと。

## ■ 手動登録時の制限事項

手動登録は自動収集（SG-AWS / SG-Azure）と異なり、以下の制限がある。これらはServiceNowの仕様上の制約であり、手動登録を行う場合は本制限を受容した上で運用する必要がある。

### 制限事項1：CI Relationships（依存関係）が自動作成されない

| 項目 | 内容 |
| :--- | :--- |
| **制限内容** | 手動登録したクラウドリソースでは、CI間の依存関係（CI Relationships）が自動作成されない。そのため、依存関係ビューによる構成の俯瞰はできない。 |
| **理由** | SG-AWS / SG-Azure による自動収集では、クラウドプロバイダーのAPIを通じてリソース間の依存関係情報を取得し、CI Relationshipsとして自動登録する。手動登録ではこのAPI取得プロセスが介在しないため、Relationshipsは作成されない。 |
| **運用上の影響** | 障害時の影響範囲確認や構成全体の俯瞰が、CMDB上の依存関係ビューからは行えない。影響範囲の確認が必要な場合は、別途管理台帳やクラウドプロバイダー側のコンソール等で補完する。 |

### 制限事項2：リージョン・アカウント階層構造が自動反映されない

| 項目 | 内容 |
| :--- | :--- |
| **制限内容** | 自動収集では、クラウドサービスアカウント配下にリージョン別でリソースが階層的に登録されるが、手動登録ではこの階層構造が自動的には構築されない。 |
| **理由** | SG-AWS / SG-Azure はAPI応答に含まれるアカウントID・リージョン情報をもとに階層構造を自動構築する。手動登録ではAPIを経由しないため、この自動構築が行われない。 |
| **運用上の影響** | CMDB上でリージョン別のリソース一覧を階層的に確認することができない。リージョン別の確認が必要な場合は、クラウドプロバイダー側のコンソールで確認する。 |

## ■ AWS / Azure 固有の注意事項と対処方法

### 1. 誤登録時の是正方法

| 項目 | 内容 |
| :--- | :--- |
| **事象** | 手動登録時に誤った情報でCIを登録してしまった場合、是正が必要となる。 |
| **通常CIとの違い** | 通常のCI（オンプレミス機器等）は変更申請により是正可能である。AWS / Azureのクラウドリソースについても、変更申請で是正可能な項目と、iNOC担当者による直接修正が必要な項目がある。【要確認】変更申請で是正できない項目があるか ※Leader確認事項② |
| **対処方法** | 変更申請で是正可能な場合：通常の変更申請フローに従い是正する。<br>変更申請で是正できない項目がある場合：iNOC担当者にて直接修正を実施する。修正対象項目と手順は「〇〇」を参照。 |

### 2. アプリケーションサービスとの紐づけ

| 項目 | 内容 |
| :--- | :--- |
| **事象** | 手動登録CIをアプリケーションサービスにどの単位で紐づけるかの方針が必要。 |
| **選択肢と条件分岐** | 以下の条件に基づき紐づけパターンを選択する。<br><br>**条件A：アカウント内のリソース数が少なく、業務影響管理の粒度が粗くてよい場合**<br>→ **パターン①：アカウント単位で紐づけ**<br>　メリット：管理が簡易、申請量が少ない<br>　制約：個別リソースの業務影響が不明確<br><br>**条件B：個別リソースごとの業務影響管理が必要な場合**<br>→ **パターン②：EC2等リソース単位で紐づけ**<br>　メリット：関係性が明確、障害時の影響特定が容易<br>　制約：申請量が多くなる<br><br>**条件C：自動収集に近い管理粒度が求められる場合**<br>→ **パターン③：親子アプリケーションサービスを作成し、子にリソース単位で紐づけ**<br>　メリット：自動収集と同等の管理が可能<br>　制約：構造が複雑、設計・維持コストが高い |
| **ISM側推奨** | 【要確認】ISM側としての推奨パターンまたはデフォルト方針を決定する必要あり ※Leader確認事項⑤ |
| **手順** | 決定した方針に基づく紐づけ手順を利用者向け手順書に追記する。 |

### 3. クラウドサービスアカウントとの紐づけ（VR/SIR影響）

| 項目 | 内容 |
| :--- | :--- |
| **事象** | 手動登録のクラウドリソースがクラウドサービスアカウントと正しく紐づかない場合、設備種別・管理グループ情報がビジネスルール（BR）により自動反映されず、VR（脆弱性管理）やSIR（セキュリティインシデント対応）のモジュールでCIが正しく関連付けられないリスクがある。 |
| **対処方法** | 手動登録時に**「親クラウドサービスアカウント」を必須入力項目**とする。<br>これにより、BRによる設備種別・管理グループの自動反映が機能し、VR/SIRでの正しい関連付けが担保される。<br><br>**具体的な手順：**<br>1. 登録対象リソースが属するクラウドサービスアカウントのCI名を確認する<br>2. ポータル申請フォームの「親クラウドサービスアカウント」フィールドに上記CI名を指定する<br>3. 登録後、設備種別・管理グループが正しく反映されていることを確認する<br><br>【要確認】上記手順でBRが正しく動作するか、実機検証が必要 ※Leader確認事項⑥ |

### 4. 登録対象範囲の定義

| 項目 | 内容 |
| :--- | :--- |
| **事象** | 手動登録の対象をどのリソース種別まで含めるかが不明確だと、案件ごとに登録粒度がばらつく。 |
| **ISM側の定義** | 手動登録の標準対象範囲を以下のとおり定義する。【要確認】以下の定義で妥当か ※Leader確認事項⑦<br><br>- **登録対象：** 仮想マシン（EC2 / Azure VM）<br>- **登録対象外（自動収集でカバー）：** VPC、サブネット、セキュリティグループ、ストレージ（S3 / Blob）等のネットワーク・ストレージ・セキュリティ関連リソース<br>- **判断が必要な場合：** 上記に該当しないリソース種別については、iNOC担当者からISMチームへ確認の上、登録要否を判断する。 |

## ■ 混在管理時の識別・フィルタ方法

自動収集CIと手動登録CIがCMDB内で混在することは前提条件となる。以下の方法で識別・フィルタを行う。

| 操作 | 方法 |
| :--- | :--- |
| **識別方法** | Discovery Source属性により識別する。<br>- 自動収集CI：Discovery Source = "ServiceGraph" 等（SG-AWS/Azureが自動設定）<br>- 手動登録CI：Discovery Source = "Manual" または空欄（手動登録時に "Manual" を設定するルールとする） |
| **CI検索時のフィルタ** | CI一覧画面にて、Discovery Sourceカラムでフィルタを適用する。<br>例：手動登録CIのみ表示 → Discovery Source = "Manual" で絞り込み |
| **レポート作成時** | レポート条件にDiscovery Sourceを含め、自動収集CI/手動登録CIを分けて集計可能とする。 |

## ■ まとめ

AWS / Azureのクラウドリソースの手動登録について、以下のとおり整理する。

- 登録対象リソース種別の標準範囲の定義
- 自動収集CI/手動登録CIの識別ルール（Discovery Source属性）とフィルタ手順
- 紐づけパターンの条件分岐と各パターンの手順（利用者向け手順書に追記）

**技術的にカバーできない範囲（各構築案件に依存）：**
- アプリケーションサービスとの紐づけパターンの最終選定は、案件の業務要件に依存するため、案件ごとに判断が必要。ISM側では条件分岐の基準と各パターンの手順を提供する。
- 案件固有の管理要件（特殊なタグ付け、独自の分類等）については、汎用項目（Attributes等）を活用して案件ごとに対応いただきたい。
